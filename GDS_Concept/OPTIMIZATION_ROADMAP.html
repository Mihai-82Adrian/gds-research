Phase 2: Resilience & Crash Recovery (CRITICAL FOR LONG RUNS)

D. LMDB Checkpointing ‚è±Ô∏è Estimated: 3-4 hours

Why: 8-10 hour processing ‚Üí crash at hour 7 = total loss

Dependencies:

heed = "0.20"  # Safe LMDB bindings

Checkpoint Data:

struct Checkpoint {
    last_processed_assertion: u64,
    last_particle_id: u64,
    particles_written: u64,
    output_file_offset: u64,
    timestamp: SystemTime,
}

LMDB Benefits:

  • Copy-on-write ‚Üí atomic commits (crash-safe)
  • Zero-copy reads (mmap)
  • Compact (few MB for metadata)

Implementation:

use heed::{Database, EnvOpenOptions};

pub struct CheckpointManager {
    env: Env,
    db: Database<OwnedType<u64>, SerdeBincode<Checkpoint>>,
}

impl CheckpointManager {
    pub fn save_checkpoint(&self, cp: &Checkpoint) -> Result<()> {
        let mut wtxn = self.env.write_txn()?;
        self.db.put(&mut wtxn, &0, cp)?;
        wtxn.commit()?;  // Atomic, crash-safe
        Ok(())
    }

    pub fn load_checkpoint(&self) -> Result<Option<Checkpoint>> {
        let rtxn = self.env.read_txn()?;
        Ok(self.db.get(&rtxn, &0)?.cloned())
    }
}

Usage Pattern:

// Load checkpoint at startup
if let Some(cp) = checkpoint_mgr.load_checkpoint()? {
    println!("Resuming from particle {}", cp.last_particle_id);
    skip_to_assertion(cp.last_processed_assertion);
}

// Save every N particles
if particles_written % 10_000 == 0 {
    checkpoint_mgr.save_checkpoint(&Checkpoint {
        last_processed_assertion: current_assertion,
        last_particle_id,
        particles_written,
        output_file_offset: file.stream_position()?,
        timestamp: SystemTime::now(),
    })?;
}

struct QualityFilter {
    min_relations: usize,
    require_embedding: bool,
    prefer_vad: bool,
    target_language_ratio: HashMap<String, f64>,
}

impl QualityFilter {
    fn score(&self, concept: &ConceptInfo) -> u32 {
        let mut score = 0;

        if concept.relations.len() >= self.min_relations {
            score += concept.relations.len() as u32;
        }

        if self.require_embedding && concept.has_embedding {
            score += 100;
        }

        if self.prefer_vad && concept.has_vad {
            score += 50;
        }

        score
    }

    fn accept(&self, concept: &ConceptInfo, current_stats: &Stats) -> bool {
        let score = self.score(concept);
        let lang_ratio = current_stats.language_ratio(&concept.language);

        score >= threshold && lang_ratio < target_ratio
    }
}

üéì Theoretical Foundations - Binary HDC

Why Binary Works

Mathematical Equivalence:

Cosine similarity (real-valued) ‚âà Hamming similarity (binary)

For normalized vectors:
  cos(A, B) = (A ¬∑ B) / (||A|| ||B||)

For binary vectors (¬±1):
  Hamming similarity = 1 - (Hamming distance / D)

Where D = dimensionality (20,000)

Information Preservation:

  • Random projection: 300D ‚Üí 20,000D (66x expansion)
  • Johnson-Lindenstrauss: distances preserved with high probability
  • Binarization: sign(x) retains directional information
  • Loss: magnitude information (acceptable for semantic similarity)

HDC Operations (Binary):

Bind:    A ‚äô B = XOR(A, B)         # Element-wise XOR
Bundle:  A ‚äï B = MAJORITY(A, B)    # Bit voting
Permute: œÅ(A) = ROTATE(A, k)       # Circular shift

Hardware Efficiency:

  • XOR: 1 CPU cycle (vs float multiply: 3-5 cycles)
  • POPCNT: Built-in instruction (SSE4.2, AVX2)
  • Memory: 64x reduction enables L1/L2 cache residence

References

Core Papers:

  1. Kanerva, ‚ÄúHyperdimensional Computing: An Introduction‚Äù (2009)
    • https://redwood.berkeley.edu/wp-content/uploads/2020/08/HDCHAPTER-1.pdf

    • Compositional semantics with distributed representations
  3. Gayler, ‚ÄúVector Symbolic Architectures‚Äù (2003)
    • Theoretical foundations of VSA/HDC
