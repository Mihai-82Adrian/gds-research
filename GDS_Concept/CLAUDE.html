CLAUDE.md

Purpose: authoritative context for every coding agent (Claude Code, Codex/GPT-5, Gemini CLI, etc.) touching this repository. Read this file before making decisions; keep it synchronized with the codebase.

1. Project Identity

  • Name: GDS – Geometrodynamic Semantics
  • Mission: Build a continuously learning multilingual semantic lexicon using hyperdimensional computing and physics-inspired semantics (concepts have mass m₀, charge q, spin s).
  • Languages: Romanian (RO), English (EN), German (DE). Additional languages must only be added after extending all pipelines and lexicons.
  • Status (2025‑10‑06): Production pipeline live. End-to-end Rust + Julia build with Parquet output and daily BabelNet enrichment. Current emphasis: scaling volume while preserving quality.

4. Build & Execution

4.1 CLI (primary entry point)

cargo run -- --config config/lexicon_build.toml [options]

Important options:

  • --max-assertions N: stream only first N ConceptNet assertions (use for dry runs).
  • Override paths: --conceptnet, --numberbatch, --vad, --german-norms, --babelnet-cache, --output.
  • Quality overrides: --min-relations, --require-embedding, --require-vad, --target-particles, --max-relations-per-concept.

Release build for production:

cargo run --release -- --config config/lexicon_build.toml

Cargo automatically rebuilds the crate when sources change; no manual rebuild step is required.

4.2 Configuration (config/lexicon_build.toml)

conceptnet_path = "data/raw/assertions.csv"
numberbatch_path = "data/raw/numberbatch.txt"

german_norms_path = "data/raw/german_norms/de_emo_norms.txt"
babelnet_cache_path = "data/enrichment/babelnet_cache.db"

require_vad = false
# 0 => unlimited in code
target_particles = 0
max_relations_per_concept = 64

[target_language_ratio]
en = 0.60

# ro omitted intentionally (no cap)

4.3 Pipeline stages (Rust LexiconBuilder::build)

  1. Load resources: Numberbatch (full), NRC VAD, German norms, BabelNet cache (if existing). Build affective axes from high/low VAD embeddings.
  2. Stream ConceptNet: ConceptNetParser::process_in_batches collapses assertions into per-concept accumulators, deduping edges and recording weights.
  3. Quality scoring: ConceptQualitySnapshot computed for each concept, scoring by relation count, embedding/VAD/BabelNet coverage. Soft language ratios include tolerance that shrinks as dataset grows.
  4. Sort & filter: Candidates sorted by score. QualityFilter::evaluate returns accept/reject + reason (telemetry); ROMANIAN has no upper cap by default.
  5. Enrichment merge: BabelNet relations appended with weight 1.2 when cache entry exists.
  6. Mass & spin: calculate_mass(frequency_prior) scales ln(1 + relation_count) × average weight with OEWM frequency boosts before assigning spins.
  7. Julia HDC: JuliaHDC::expand_to_20kd communicates with persistent Julia server. Non-embedded concepts fall back to zero vector (keeps density at ~0.5).
  8. Parquet export: ParticleWriter::write_particle buffers into batches (1k). Metadata collects unique counts and file size. The enriched schema now includes provenance columns (canonical_id, alias_primary, alias_candidates_json, frequency_prior, embedding_source, affective_source, babelnet_id, source_flags_json, relations_json) beside the numeric mass/spin/vector fields.
  9. Report: CLI prints number of concepts aggregated, candidate summary (percentage with embeddings/VAD/BabelNet), particles written, coverage, language distribution.

Observation: With min_relations = 3 and require_embedding = true, expect roughly 60–70% of candidates to remain; VAD coverage is ~35–45% on ConceptNet subsets. Tune thresholds accordingly.

5. Quality Heuristics & Monitoring

Lever Effect Typical Setting
min_relations Minimum unique relations to keep concept 3 (raise to 4–5 for higher precision)
require_embedding Discard concepts lacking Numberbatch vector true in production
require_vad Force VAD coverage false (use spin fallback)
target_particles Soft cap. 0 means unlimited Use for staged builds (e.g., 100_000)
target_language_ratio Soft cap per language Adjust to steer distribution; tolerance auto-adjusted
max_relations_per_concept Prevent huge adjacency lists 64 (tune to reduce Parquet size)

Diagnostic command after build:

cargo run --example analyze_lexicon_quality -- data/processed/full_lexicon.parquet

Prints total particles, per-language counts, mass percentiles, bit density, and valence distribution.

6. Daily Enrichment Loop

  1. scripts/daily_enrichment.sh

    • Sources .config/babelnet.env (must contain export BABELNET_API_KEY="...").
    • Executes cargo run --release --example enrich_daily_batch (adaptive batch size, dedupes using cache, stops when BabelCoins low).
    • On success: cargo run --release -- --config config/lexicon_build.toml to regenerate Parquet.
    • Weekly Sunday backups compressed (gzip), 8-week retention.

  2. scripts/setup_cron.sh installs a cron entry at 02:00 daily. Re-run script after editing the shell script.

  3. Priority queue builder (examples/build_enrichment_queue.rs) ranks top BabelNet targets. Ensure it is refreshed periodically if ConceptNet scoring parameters change.

7. Testing

  • cargo test --lib & cargo test are required to pass. Julia integration test auto-skips (with warning) if Julia executable/config is missing—never ignore failures silently.
  • Integration checks:
    • cargo run --example test_parquet_writer
    • cargo run --example test_julia_integration
    • cargo test --test mini_build (mini pipeline with mock Julia + fixtures)
    • cargo run --example analyze_lexicon_quality -- <parquet>
  • Before large builds, run cargo check --release to catch unused imports/typos.

9. Agent Operating Rules (Non-negotiable)

  1. Quality First: zero tolerance for mock-ups, stubs, placeholder logic, hard-coded demos, commented-out code to “keep for later,” or partially implemented features left surfaced.
  2. Sequential Reasoning: before implementing any complex or novel feature, run the sequential-thinking MCP tool to outline approach, and consult the web search MCP for external references as needed.
  3. Documentation: create/append an annex (e.g., docs/<topic>.md) for any significant change, summarizing research and decisions. Reference the annex from this file when merging major features.
  4. No Warning Suppression: compiler warnings or runtime errors must be resolved through proper fixes, never hidden or bypassed with dummy values or allow attributes.
  5. Testing Discipline: run cargo fmt, cargo check, and relevant tests before marking work complete. For Julia-dependent code, ensure the integration test path succeeds or explicitly documents environment limitations.
  6. Config Consistency: if configuration schema changes, update config/lexicon_build.toml, this file, and the CLI parser simultaneously.
  7. Dependency Changes: record library upgrades or new dependencies in a short docs/changes/<date>_dependency_update.md and update this file’s relevant sections.
  8. Review Mindset: treat every code change as production-grade. Evaluate readability, performance, and scalability. If uncertain, stop and research before coding.
  9. Communication: if encountering ambiguous requirements, pause and clarify through repo notes or direct questions—no assumptions.

These rules apply to all AI agents and human contributors. Violations must be corrected immediately.
